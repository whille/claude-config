# Refactoring Rules

> Martin Fowler《Refactoring》核心原则，浓缩为 Claude 可执行规则

---

## 核心原则

### 重构定义

**在不改变代码外部行为的前提下，改善其内部结构。**

```
重构前：功能 A
重构后：功能 A（行为不变）+ 结构更好
```

---

## 重构前检查

| 检查项 | 说明 |
|--------|------|
| 有测试吗？ | 没测试 = 不能重构 |
| 行为明确吗？ | 不清楚原有行为 = 停止 |
| 能 import 吗？ | 能 import = 不要复制 |

**无测试代码的处理**：
1. 先写测试（至少记录当前行为）
2. 再重构

---

## 三大红线

| 红线 | 触发条件 | 说明 |
|------|----------|------|
| **行为变更** | 输入输出改变 | 立即停止 |
| **代码膨胀** | 新增 > 10% | 方向错误 |
| **批量改动** | > 1 个文件 | 步子太大 |

---

## 正确做法

### 提取方法 (Extract Method)

```
# 错误：复制粘贴
def new_func():
    # 复制 old_func 的代码
    ...

# 正确：导入复用
from module import old_func

def new_func():
    old_func()  # 调用
```

### 移动方法 (Move Method)

```
# 错误：在新文件重写
new_module.py:
    def method():  # 重新实现
        ...

# 正确：导入后委托
new_module.py:
    from old_module import method
    # 或创建别名
```

### 小步重构

```
Step 1: 改一个函数 → 测试 → 提交
Step 2: 改另一个 → 测试 → 提交
Step 3: ...

禁止：改10个函数 → 测试（出问题不知道哪一步错）
```

---

## 坏味道检测

| 坏味道 | 解决方案 |
|--------|----------|
| 重复代码 | 提取共用函数 |
| 过长函数 | 拆分子函数 |
| 过大类 | 拆分职责 |
| 发散式变化 | 一个类多个变化点 → 拆分 |
| 霰弹式修改 | 一个变化改多处 → 合并 |

---

## 重构清单模板

重构前必须填写：

```markdown
## 重构计划

**目标**: 一句话说明目的（如：统一入口）

**可复用函数**:
- `module.func_a` → 直接 import
- `module.func_b` → 直接 import

**新增代码**:
- `signal_strategy.py` → 入口调度（预估 200 行）

**红线检查**:
- [ ] 新增 < 10%
- [ ] 报告格式不变
- [ ] 每步测试
```

---

## 代码异味判断

```
需要复制粘贴代码？
  → 错了，应该 import

新增文件 > 1？
  → 步子太大，先改入口

原有代码"不好"想重写？
  → 先复用，后续单独优化
```

---

## 验证标准

重构完成后：

| 检查项 | 方法 |
|--------|------|
| 行为一致 | 相同输入 → 相同输出 |
| 测试通过 | pytest / 原有测试 |
| 代码更少 | wc -l 对比 |
| 职责清晰 | 一个文件一件事 |
