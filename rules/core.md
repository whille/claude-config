# Core Rules

> 核心开发规范，全局生效

---

## 代码质量标准

| 指标 | 限制 |
|------|------|
| 函数长度 | < 50 行 |
| 文件长度 | < 800 行 |
| 嵌套层级 | ≤ 4 层 |
| 测试覆盖率 | ≥ 80% |

---

## 核心原则

### 简单性优先

**最小代码解决问题。无投机性功能。**

- 不写超出需求的功能
- 不为单次使用创建抽象
- 如果 50 行能解决，不写 200 行

### 不可变性

创建新对象，不修改现有对象。

### 错误处理

- 每层显式处理
- 绝不静默吞掉错误

### 输入验证

- 系统边界验证
- 快速失败
- 不信任外部数据

---

## 工作流

1. **编码前**：明确假设，不隐藏困惑，有多种理解时全部呈现
2. **研究先行**：GitHub → 官方文档 → 搜索
3. **TDD**：先写测试 → 实现 → 重构
4. **审查**：写完代码立即 code-reviewer agent
5. **提交**：Conventional Commits 格式

### 成功标准

| 任务 | 转化为 |
|------|--------|
| "添加验证" | 写测试覆盖无效输入 → 通过 |
| "修复 Bug" | 写测试复现 → 通过 |
| "重构 X" | 测试前后都通过 |

---

## 安全检查

以下场景**立即**使用 security-reviewer：

- 认证/授权代码
- 用户输入处理
- 数据库查询
- 外部 API 调用

**常见问题**：硬编码凭证、SQL 注入、XSS

## Git 规范

```
<type>: <description>

Types: feat | fix | refactor | docs | test | chore | perf
```

- 不添加 `Co-Authored-By: Claude`
- 分支命名：`feat/xxx`, `fix/xxx`

---

## 编码风格

### 注释语言

关键逻辑用中文注释，代码标识符用英文。

### 命名风格

| 语言 | 惯例 |
|------|------|
| Python | `snake_case` |
| JS/TS | `camelCase` |
| 常量 | `UPPER_SNAKE_CASE` |

---

## 交互习惯

- **高风险操作确认**：删除、推送、重置前必须确认
- **输出风格**：带文件路径引用 `file_path:line_number`
- **问题处理**：先分析根因，不确定时先问用户

---

## 编辑前必须 Read

在执行 Edit 或 Write 之前：

1. **必须先 Read 目标文件**（除非是新建文件）
2. **必须检查 system-reminder 的文件修改提示**
3. **必须用最小范围编辑**（Edit 优于 Write）

**禁止**：
- 不读取就编辑现有文件
- 用 Write 覆盖整个文件
- 改变原有格式

---

## 常见陷阱

| 陷阱 | 错误示例 | 正确做法 |
|------|----------|----------|
| Falsy 陷阱 | `if config:` | `if config is not None:` |
| 日期切片 | `arr[-N:]` | `filter(d >= cutoff)` |
| 空字典默认 | `func(x={})` | `func(x=None)` |

**提交前必验证**：打印实际日期范围和数据量。
