# Workflow Rules

> 工作流与调用追踪规范

---

## 调用链追踪

### 核心原则

**从目标函数向上追踪，而不是从关键字猜测。**

### 正确方法

| 场景 | 方法 |
|------|------|
| 找"谁生成了某文件" | LSP `incomingCalls` 从写入函数向上追踪 |
| 找"谁调用了某函数" | LSP `incomingCalls` |
| 找"某函数调用了谁" | LSP `outgoingCalls` |

### 禁止做法

- ❌ grep 搜关键字 → 猜测调用链
- ❌ 看到"读取"函数就以为是入口
- ❌ 只看一层调用就下结论

---

## 回答前验证

当用户问以下问题时，**先验证再回答**：

- "这个函数做什么的？"
- "当前配置是什么？"
- "参数值是多少？"

**正确做法**：
1. Read 文件确认
2. 引用具体位置 `file_path:line_number`
3. 不确定时声明"让我验证一下"

---

## 深度审查触发

以下变更**建议执行** `/deep-review`：

### 高风险文件

- `**/auth*`、`**/security*`
- `**/password*`、`**/token*`
- `**/payment*`、`**/user*`

### 大规模变更

- 修改文件数 ≥ 5
- 变更行数 ≥ 200

### 安全敏感操作

- 用户输入处理
- API 端点新增/修改
- 数据库操作
- 外部 API 调用

---

## 重构建议触发

代码变更完成后，检测以下问题并建议：

| 问题 | 检测规则 | 建议 |
|------|----------|------|
| 重复代码 | 相同代码块 ≥ 5 行 | `/simplify` |
| 函数过长 | 函数 > 50 行 | "拆分这个函数" |
| 嵌套过深 | 嵌套 > 4 层 | "降低嵌套" |
| 参数过多 | 参数 > 5 个 | "封装参数" |

---

## 完成前验证

**在声称完成前，必须运行验证命令**：

```bash
# 测试
pytest tests/

# 覆盖率
pytest --cov=src

# 类型检查（如有）
mypy src/

# Lint（如有）
ruff check src/
```

**证据先于断言**：确认输出后再声称成功。
