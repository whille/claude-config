# Refactoring Suggestions

> 代码重构建议规则 - 检测问题并提供建议，不自动执行

---

## 触发时机

代码变更完成后，检测以下问题：

---

## 检测清单

### L1: 适合 `/simplify` 的问题

| 问题 | 检测规则 | 建议 |
|------|----------|------|
| **重复代码** | 相同/相似代码块 ≥ 5 行 | "发现重复代码，建议：`/simplify` 或告诉我'消除重复'" |
| **函数过长** | 函数 > 50 行 | "函数过长({n}行)，建议：`/simplify` 或告诉我'拆分这个函数'" |
| **嵌套过深** | 嵌套层级 > 4 | "嵌套过深({n}层)，建议：`/simplify` 或告诉我'降低嵌套'" |
| **命名不清** | 单字母变量、无意义缩写 | "命名不清晰，建议：`/simplify` 或告诉我'改善命名'" |
| **参数过多** | 函数参数 > 5 个 | "参数过多({n}个)，建议：`/simplify` 或告诉我'封装参数'" |

### L2: 适合直接指令的问题

| 问题 | 检测规则 | 示例指令 |
|------|----------|----------|
| **类过大** | 文件 > 800 行 | "文件过大({n}行)，建议：告诉我'拆分这个类'" |
| **循环依赖** | A → B → C → A | "发现循环依赖，建议：告诉我'修复循环依赖'" |
| **违反 SRP** | 一个类做多个事 | "职责不单一，建议：告诉我'按职责拆分'" |
| **缺少设计模式** | 大量 if-else/switch | "适合策略模式，建议：告诉我'用策略模式重构'" |
| **紧耦合** | 直接实例化依赖 | "紧耦合问题，建议：告诉我'引入依赖注入'" |

## 输出格式

检测到问题时，输出：

```
📊 检测到重构机会：

【文件】src/user_manager.py

├─ 函数过长：process_user() 120行
│  💡 建议：/simplify 或 "拆分这个函数"
│
├─ 重复代码：第45-60行 与 第80-95行
│  💡 建议：/simplify 或 "消除这段重复代码"
│
└─ 嵌套过深：validate() 有5层嵌套
   💡 建议：/simplify 或 "用early return降低嵌套"

💡 小贴士：直接描述你的意图，如"把UserManager拆成三个类"
```

## 设计哲学

### Rule vs Skill 的边界

```
Rule = 检测 + 建议（被动）
Skill = 执行（主动触发）

✅ Rule: 发现问题 → 输出建议 → 等用户决定
❌ Rule: 发现问题 → 自动调用 skill
```

### 为什么不自动执行

1. **重构有风险**：结构性变更影响广，需要人类判断
2. **上下文重要**：同样的代码在不同场景有不同最佳实践
3. **渐进式改进**：用户可能只想解决部分问题
4. **学习机会**：让用户理解问题所在

## 红线检测

以下情况**强烈建议**用户处理：

```
🚨 必须关注：
- 安全相关文件
- 认证/授权代码（auth*, permission*）
- 数据库操作（SQL拼接）
- 用户输入处理

这些变更建议执行 /deep-review
```

---

## 最佳实践

### 优先级排序

```
P0: 安全问题（立即处理）
P1: 架构问题（计划处理）
P2: 代码质量（有空处理）
P3: 风格问题（可选）
```

### 渐进式重构

```
❌ 不要：一次性重构整个模块
✅ 应该：小步重构 → 测试 → 提交 → 继续

"重构是整理代码，不是重写代码"
```

## 配置选项

用户可在项目 `.claude/rules/` 中覆盖：

```markdown
# 项目特定规则

## 跳过检测
- tests/**（测试文件不做重构建议）
- legacy/**（遗留代码暂不处理）

## 调整阈值
- 函数长度阈值：50 → 80（项目允许较长函数）

## 强制检测
- src/core/**（核心模块必须检测）
```
