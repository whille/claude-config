# Deep Review

深度多维度代码审查，使用并行 subagents 全面分析代码。

last_updated: 2026-04-19
---

## 与原生 /review 的区别

| 功能 | 原生 /review | /deep-review |
|------|--------------|--------------|
| 审查维度 | 单一维度 | 4维度并行（安全+质量+性能+架构） |
| 执行方式 | 串行 | 并行 subagents |
| 交互性 | 有 | 增强交互（分维度确认） |
| 输出格式 | 标准 | 分维度报告 + 汇总 |
| 修复建议 | 有 | 有 + 自动生成代码 |

## 四维度审查内容

### 1. 安全审查

| 检查项 | 说明 |
|--------|------|
| **注入漏洞** | SQL/Command/Code Injection |
| **认证授权** | Auth bypass, 权限控制 |
| **敏感数据** | 密码/API Key 硬编码 |
| **XSS/CSRF** | 跨站脚本/请求伪造 |
| **输入验证** | 用户输入过滤 |

### 2. 质量审查

| 检查项 | 说明 |
|--------|------|
| **SOLID 原则** | 单一职责、开闭原则等 |
| **代码复杂度** | 圈复杂度、嵌套深度 |
| **命名规范** | 变量/函数命名 |
| **重复代码** | DRY 原则 |
| **测试覆盖** | 缺失测试 |

### 3. 性能审查

| 检查项 | 说明 |
|--------|------|
| **算法复杂度** | O(n²) 以上警告 |
| **资源泄漏** | 未关闭连接/文件 |
| **循环优化** | 循环内重复计算 |
| **缓存机会** | 可缓存未缓存 |
| **N+1 查询** | 数据库查询问题 |

### 4. 架构审查

| 检查项 | 说明 |
|--------|------|
| **依赖关系** | 循环依赖、过耦合 |
| **模块边界** | 职责划分不清 |
| **设计模式** | 模式滥用/误用 |
| **扩展性** | 难以扩展的设计 |
| **一致性** | 风格不统一 |

## 使用方式

### 基本使用

```bash
# 审查当前变更（git diff）
/deep-review

# 审查指定文件
/deep-review src/main.py

# 审查目录
/deep-review src/
```

### 交互流程

```
1. 分析变更范围
   ├── 文件数: 5
   ├── 变更行数: 120
   └── 风险级别: 中

2. 并行执行 4 维度审查...

3. 分维度展示结果：

📊 安全审查（发现 2 个问题）
┌────────────────────────────────────────────┐
│ 🔴 P0: SQL注入风险                          │
│    文件: src/db.py:45                       │
│    问题: 用户输入直接拼接到 SQL                 │
│    修复: 使用参数化查询                      │
└────────────────────────────────────────────┘

是否修复这些问题？[y/n/select]

📊 质量审查（发现 1 个问题）
...

4. 用户选择修复维度

5. 生成修复代码并应用
```

## 完整工作流示例

### 场景：审查发现架构问题

```
用户：/deep-review src/

Claude：
## Deep Review 报告

### 执行摘要

| 维度 | 问题数 | 严重 | 高 | 中 | 低 |
|------|--------|------|----|----|----|
| 安全 | 0 | 0 | 0 | 0 | 0 |
| 质量 | 2 | 0 | 1 | 1 | 0 |
| 性能 | 1 | 0 | 0 | 1 | 0 |
| 架构 | 1 | 1 | 0 | 0 | 0 |

### 🏗️ 架构问题（P0）

**问题**：UserManager 职责混乱

| 属性 | 值 |
|------|-----|
| 文件 | src/user_manager.py (850行) |
| 问题 | 包含认证、资料、权限三种职责 |
| 影响 | 难以维护、测试、扩展 |

### 📊 Phase 5: 后续行动建议

根据问题分类，建议以下步骤：

| 问题类型 | 建议工具 | 说明 |
|----------|----------|------|
| 🏗️ 架构问题 | `/refactor --plan` | 需要架构级重构 |
| 📝 质量问题 | `/simplify` | 细粒度改进 |
| ⚡ 性能问题 | 直接优化 | 优化算法复杂度 |

**推荐执行顺序**：

```
1. 先修复安全问题（如果有）✅ 无

2. 执行架构重构（优先级高）
   /refactor src/user_manager.py --plan

3. 架构重构完成后，执行 /simplify 清理代码

4. 最后处理性能优化
```

是否执行架构重构方案？[Y/n]

用户：Y

Claude：启动 /refactor...
（进入 /refactor 流程）
```

## 配置选项

可在 `$HOME/.claude/config/deep-review.yaml` 配置：

```yaml
# 审查级别
level: "standard"  # quick | standard | deep

# 启用的维度
dimensions:
  security: true
  quality: true
  performance: true
  architecture: true

# 并发数
concurrency: 4

# 自动修复
auto_fix: false  # true 则不询问直接修复

## 与 /refactor 的协作流程

### 问题分类决策树

```
审查发现问题
    │
    ├─ 安全问题？
    │   └─ 直接修复（参数化查询、移除硬编码等）
    │
    ├─ 性能问题？
    │   ├─ 简单优化 → 直接修复（加缓存、改算法）
    │   └─ 复杂优化 → 输出优化方案
    │
    ├─ 质量问题？
    │   ├─ 重复代码/命名 → 建议 /simplify
    │   └─ 复杂度/嵌套 → 建议 /simplify
    │
    └─ 架构问题？
        │
        ├─ 需要重构的信号：
        │   - 文件 > 500行 且 职责混乱
        │   - 循环依赖 (A → B → C → A)
        │   - 违反 SRP 且 需要 Extract Class
        │   - 模块边界不清
        │   - 过耦合（改一处影响多处）
        │
        └─ 输出建议：
            "📊 检测到架构级问题：
             - UserManager: 800行，职责混合（认证+资料+权限）
             - 建议：执行 /refactor skills/user_manager.py --plan
             "
```

### 触发 /refactor 的阈值

| 指标 | 阈值 | 说明 |
|------|------|------|
| 文件行数 | > 500行 | 可能需要拆分 |
| 类行数 | > 300行 | 建议职责分离 |
| 循环依赖 | 存在 | 必须修复 |
| 圈复杂度 | > 15 | 函数过长 |
# 忽略规则
ignore:
  - "tests/**"
  - "**/*.md"
```
